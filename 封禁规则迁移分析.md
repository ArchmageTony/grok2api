# 封禁规则覆盖分析

## 旧架构（at-mod `manager.py`）的自定义规则

旧代码中定义了 `_INVALID_ACCOUNT_RULES` 规则表，命中时将 Token 标记为**失效（expired）**：

| HTTP 状态码 | 关键词 | 标记原因 |
|------------|--------|----------|
| 403 | `blocked-user` | `rate_limits_blocked_user_403` |
| 403 | `unauthorized:blocked-user` | `rate_limits_blocked_user_403` |
| 403 | `bot abuse` | `rate_limits_blocked_user_403` |
| **400** | **`email-domain-rejected`** | `rate_limits_email_domain_rejected_400` |

---

## 新架构的覆盖情况

### ✅ 已覆盖：`invalid-credentials`（400）

**文件**：`app/dataplane/reverse/protocol/xai_usage.py`

```python
def is_invalid_credentials_body(body: str) -> bool:
    text = str(body or "").lower()
    return "invalid-credentials" in text or "failed to look up session id" in text
```

当响应体含 `invalid-credentials` 或 `failed to look up session id` 时，分类为 `AUTH_FAILURE`，并在 `invalid_credentials.py` 中将账户标记为 `EXPIRED`。

---

### ✅ 已覆盖：403 Forbidden → DISABLED

**文件**：`app/control/account/state_machine.py`

```python
elif feedback.kind == FeedbackKind.FORBIDDEN:
    strikes = int(ext.get(_FORBIDDEN_STRIKE_KEY, 0)) + 1
    ext[_FORBIDDEN_STRIKE_KEY] = strikes
    if strikes >= policy.forbidden_strikes:  # 默认 1 次
        status = AccountStatus.DISABLED
```

403 响应（包括 `blocked-user`、`bot abuse` 等情况）会通过 `FeedbackKind.FORBIDDEN` 进入此逻辑，1 次即触发 `DISABLED`。

但注意：`classifier.py` 中对 403 的处理有区分：
- 含 CF Challenge 关键词 → `ResultCategory.FORBIDDEN`（代理失效）
- 其他 403 → `ResultCategory.AUTH_FAILURE`（直接账户失效）

---

### ❌ **缺口：`400 email-domain-rejected`**

**旧架构**：明确匹配 `400` + `email-domain-rejected` 关键词 → 标记为 `EXPIRED`

**新架构**：`is_invalid_credentials_body()` 仅匹配：
- `invalid-credentials`
- `failed to look up session id`

**不包含** `email-domain-rejected`，因此 `400 email-domain-rejected` 在新架构中会**被当作普通错误忽略**，不会触发账户标记失效。

---

### ❌ **缺口：`403 bot abuse`（部分）**

`classifier.py` 中 403 分支逻辑：
```python
if status_code == 403:
    if body and ("cf-challenge" in body.lower() or "cloudflare" in body.lower()):
        return ResultCategory.FORBIDDEN
    return ResultCategory.AUTH_FAILURE  # ← bot abuse 走这里
```

`bot abuse` 走 `AUTH_FAILURE`，但 `AUTH_FAILURE` 在状态机中会触发哪个 `FeedbackKind` 还需要确认调用方如何映射——**大概率**等同于 `UNAUTHORIZED`（401 逻辑），而非 `FORBIDDEN`（不触发 DISABLED strike）。

---

## 建议

### 方案 A：扩展 `is_invalid_credentials_body()`（推荐）

在 `app/dataplane/reverse/protocol/xai_usage.py` 中添加 `email-domain-rejected` 的识别：

```python
def is_invalid_credentials_body(body: str) -> bool:
    text = str(body or "").lower()
    return (
        "invalid-credentials" in text
        or "failed to look up session id" in text
        or "email-domain-rejected" in text   # ← 新增
    )
```

效果：400 email-domain 错误 → `is_invalid_credentials_error()` 返回 True → 账户标记 `EXPIRED`，与旧逻辑一致。

### 方案 B：验证 `bot abuse` / `blocked-user` 的 403 处理链路

检查 `feedback.py` 或调用 `classify_result()` 的上层代码，确认 `AUTH_FAILURE` → 实际传入 `state_machine` 的 `FeedbackKind` 是 `UNAUTHORIZED` 还是 `FORBIDDEN`，以确认 `blocked-user` 是否能正确触发 DISABLED。
